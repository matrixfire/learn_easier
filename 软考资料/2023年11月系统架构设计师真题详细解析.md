## 二、案例分析题详细解析

### 试题一：大数据架构（Lambda vs Kappa）

**背景**：某网站奥运期间大规模视频观看数据的实时分析需求。

**问题1（8分）-- Lambda架构与Kappa架构对比填表**：

| 对比内容 | Lambda架构 | Kappa架构 |
|----------|-----------|-----------|
| 复杂度与开发维护成本 | 需维护**2**套系统，复杂度**高** | 只需维护**1**套系统，复杂度**低** |
| 计算开销 | 批处理+实时计算一直运行，开销大 | 必要时全量计算，开销**相对较小** |
| 实时性 | 满足实时性 | **满足实时性** |
| 历史数据处理 | 批式全量处理，吞吐量**大** | 流式全量处理，吞吐量较低，能力**弱** |

**问题2（9分）-- 大数据架构技术选型填空**：

| 空号 | 答案 | 说明 |
|------|------|------|
| (1) | Spark(d) | 离线计算引擎 |
| (2) | MapReduce(e) | 离线计算引擎 |
| (3) | 数据存储层(k) | 层级分类 |
| (4) | MemSQL(g) | 实时数据库 |
| (5) | HDFS(h) | 分布式文件系统 |
| (6) | 数据采集层(l) | 层级分类 |
| (7) | Flume(j) | 日志采集 |
| (8) | ETL(f) | 数据集成 |
| (9) | 业务逻辑层(n) | 层级分类 |

**问题3（8分）-- Lambda架构三层**：
- (1) 批处理层
- (2) 加速层
- (3) 服务层
- 该系统基于**Lambda架构**搭建

**考点总结**：大数据架构是近年热点，Lambda与Kappa的对比必须掌握，同时需要熟悉Hadoop/Spark生态各组件的定位。

### 试题二：SysML需求图 vs UML用例图

**问题1**：
- **需求图(REQ)**：用于表述文字化需求、需求间关系，以及与之存在满足/验证关系的模型元素。偏向系统内部功能和流程。
- **用例图**：从用户角度提供系统或业务流程功能概述。偏向用户使用体验和系统交互。

**问题2**：需求图七种关系
1. **复合关系** -- 包含子需求的层次结构
2. **派生关系** -- 从上层需求派生下一级需求
3. **细化关系** -- 用模型元素进一步细化需求
4. **满足关系** -- 设计/实现模型如何满足需求
5. **验证关系** -- 测试用例验证需求
6. **复制关系** -- 跨产品系列重用需求
7. **追溯关系** -- 需求与源文档的通用关联

**考点总结**：SysML是近年新增考点，需求图是SysML的核心新增图，七种关系必须全部掌握。

### 试题三：数据库主从复制、读写分离 + Redis缓存

**核心知识点**：
- 数据库主从复制原理（异步复制/半同步复制）
- 读写分离架构设计（主库写、从库读）
- Redis缓存数据库的应用场景与策略

### 试题四：Hibernate架构 + JWT

**Hibernate体系架构**：
- 持久化对象 -> Session -> SessionFactory -> 数据源
- 数据持久层的核心概念与配置

**JWT（JSON Web Token）优缺点**：

| 优点 | 缺点 |
|------|------|
| 无状态，易于分布式扩展 | 载荷信息不能太多 |
| 通过密钥签名保证完整性和安全性 | 安全性依赖密钥保护 |
| 支持跨域使用 | 一旦生成无法撤销 |
| 简单易用，易维护 | 需设置短期过期时间弥补无法撤销问题 |

**核心知识点**：

- 技术选型

### 论文试题二：面向对象分析

**核心框架**：
- **用例模型**：识别参与者 -> 合并需求获得用例 -> 细化用例描述 -> 调整用例模型
- **用例关系**：包含、扩展、泛化
- **用例描述**：名称/简要说明/事件流/非功能需求/前置条件/后置条件/扩展点/优先级
- **分析模型**：定义概念类 -> 识别类间关系 -> 为类添加职责 -> 建立交互图
- **类间关系**：关联、聚合、组合、泛化、依赖、实现

### 论文试题三：多数据源集成

**三种多源数据集成技术**：

| 技术 | 侧重点 | 应用场景 |
|------|--------|----------|
| 联邦式 | 数据源自治，查询时集成 | 数据源异构、实时性要求高 |
| 中间件模型 | 中间层统一接口 | 需要透明访问多数据源 |
| 数据仓库 | ETL后集中存储 | 历史数据分析、决策支持 |

### 论文试题四：边云协同

**六种协同模式**：

| 协同类型 | 边缘侧 | 云端 |
|----------|--------|------|
| 资源协同 | 提供基础设施资源，本地调度管理 | 资源调度策略下发 |
| 数据协同 | 数据采集与初步处理 | 海量数据存储、分析与价值挖掘 |
| 智能协同 | 按AI模型执行推理 | AI集中式模型训练，下发模型 |
| 管理调度 | -- | 应用开发测试环境、生命周期管理 |
| 业务管理协同 | 模块化微服务应用实例 | 按需业务编排 |
| 服务协同 | 按策略实现ECSaaS服务 | SaaS服务分布策略与能力 |

**核心思想**：云计算擅长全局性/非实时/长周期大数据处理；边缘计算擅长局部性/实时/短周期数据处理。两者互补协同。

## 五、备考建议

### 重点突破方向

1. **软件架构是绝对核心**（占比超1/4）
   - 必须熟练掌握：架构风格分类（数据流/调用返回/独立构件/虚拟机/仓库）、ATAM/SAAM评估方法、ABSD六阶段、质量效用树
   - 英文术语不能丢分：层次架构的 abstract/partition/visible 等关键词

2. **构件技术是第二大考点**
   - 构件的六个特性（可组装性/可部署性/可替换性等）
   - 构件管理三大活动
   - 适应性构件 vs 装配性构件的区别

3. **数据库范式年年考**
   - 1NF~4NF各自消除的依赖类型必须滚瓜烂熟
   - 传递依赖 -> 不满足3NF；多值依赖 -> 需要4NF

4. **SysML需求图是近年新增重点**
   - 七种关系必须全部记住（复合/派生/细化/满足/验证/复制/追溯）
   - 与UML用例图的区别

### 案例分析备考

- **大数据架构**（Lambda/Kappa）已连续多年考查，对比表格必须背熟
- **新技术趋势**：数字孪生、边云协同、JWT认证等是近年热点
- **架构填空题**：需要熟悉Hadoop/Spark/Flink生态各组件定位

### 论文备考

- 必须准备**5个以上**项目素材，覆盖不同领域
- 推荐准备方向：
  1. 可靠性设计（串联/并联模型、可靠性增长）
  2. 面向对象分析（用例模型+分析模型完整流程）
  3. 多数据源集成（联邦/中间件/数据仓库三种模式）
  4. 边云协同（六种协同模式）
  5. 大数据架构（Lambda/Kappa）
- 论文结构：项目背景(300字) + 理论阐述(400字) + 实践详述(1200字) + 总结反思(200字)

### 易错点提醒

| 易错点 | 正确答案 | 常见误选 |
|--------|----------|----------|
| McCabe圈复杂度公式 | V(G) = m - n + **2** | m - n + 1 |
| McCabe最大圈复杂度 | <= **10** | <= 20 |
| SSL不具备的安全特性 | 不可抵赖性 | 不可篡改性 |
| 管道-过滤器架构 | 管道节点**顺序**处理 | 并行处理 |
| 构件外部可见状态 | 构件**有**外部可见状态 | 没有外部可见状态 |
| 进程vs线程 | 进程=资源分配最小单位，线程=调度最小单位 | 混淆 |
| 专利申请人与发明人 | 不一定相同 | 一定是同一人 |
| 区块链分类 | 公有链/联盟链/私有链 | 行业链 |
| 局域网特征 | 封闭性网络 | 非封闭性网络 |
| ABSD需求来源 | 开发人员商业目标 | 公司经济目标 |
