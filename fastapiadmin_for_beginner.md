# FastapiAdmin 项目新手指南

> 写给刚学 Python 不久、即将参与本项目开发的你。这篇文章会用大白话帮你建立整体认知，之后再读原始架构文档就不会那么蒙了。

---

## 这个项目是干什么的？

想象一家公司需要一个**后台管理系统**——管理员登录后，可以管理员工账号、分配权限、查看日志、配置定时任务等等。FastapiAdmin 就是这个系统的**后端**（负责处理数据的那一部分）。

你可以把它理解成一个"大管家"：
- 有人要登录 → 大管家查验身份
- 有人要查数据 → 大管家检查权限后给数据
- 有人要改数据 → 大管家记录操作日志后执行修改

## 几个你必须先知道的概念

| 概念 | 一句话解释 | 生活类比 |
|------|-----------|---------|
| **FastAPI** | Python 的 Web 框架，帮你快速写 API 接口 | 餐厅的前台，负责接待客人、把点单转给厨房 |
| **SQLAlchemy** | ORM 框架，用 Python 代码操作数据库，不用写 SQL | 翻译官，帮你把 Python 话翻译成数据库话 |
| **Redis** | 内存数据库，读写极快，用于缓存和临时数据 | 便签纸，随时记随时看，关机就没了 |
| **JWT** | 一种身份令牌，登录后服务器发给你，之后每次请求带上 | 游乐园手环，入园时发你，之后凭手环玩每个项目 |
| **Alembic** | 数据库迁移工具，记录表结构的每一次变更 | 修改记录本，每次改了数据库结构都记下来，方便回溯 |
| **异步 (async/await)** | 不等一件事情做完就去做下一件，提高效率 | 烧水时不用盯着，可以去切菜，水开了再回来 |

## 项目长什么样？

```
backend/
├── main.py              ← 入口：从这里启动整个项目
├── app/                 ← 核心代码都在这里
│   ├── api/v1/          ← API 接口（对外暴露的"功能点"）
│   ├── core/            ← 框架层：数据库、权限、中间件等基础设施
│   ├── common/          ← 公共定义：常量、枚举、响应格式
│   ├── config/          ← 配置：数据库地址、密钥等
│   ├── plugin/          ← 插件：AI聊天、定时任务、代码生成器
│   ├── utils/           ← 工具函数：验证码、Excel、加密等
│   └── scripts/         ← 初始化脚本：建表、写入初始数据
├── env/                 ← 环境配置模板（开发/生产各一份）
└── tests/               ← 测试代码
```

**核心认知**：你 90% 的工作都会在 `app/` 目录下，尤其是 `api/v1/`（写接口）和 `plugin/`（写插件）。

## 一个请求进来后发生了什么？

这是最重要的部分——理解了这条链路，你就知道代码是怎么跑起来的。

```
用户点击"查看用户列表"
        │
        ▼
  ① CORS 中间件      → 检查跨域：这个请求是不是来自允许的网站？
        │
        ▼
  ② 请求日志中间件    → 记录请求信息；如果是演示模式，禁止修改操作
        │
        ▼
  ③ 限流器           → 你请求太频繁了吗？10秒内最多5次
        │
        ▼
  ④ Token 验证       → 你的登录令牌有效吗？去 Redis 查一下
        │
        ▼
  ⑤ 权限校验         → 你有"查看用户列表"的权限吗？
        │
        ▼
  ⑥ 业务处理         → Controller → Service → CRUD（查数据库）
        │                               │
        │                          ⑥.1 数据权限过滤
        │                          （你只能看自己部门的数据？还是全部？）
        │
        ▼
  ⑦ 返回结果          → 统一格式：{ code: 0, msg: "成功", data: [...] }
```

**类比**：就像进公司大楼——
1. 门卫查你是不是本楼的人（CORS）
2. 前台登记来访（日志）
3. 闸机防止拥挤（限流）
4. 刷工牌进楼（Token 验证）
5. 刷卡只能进你有权限的楼层（权限校验）
6. 到工位干活（业务处理）
7. 出楼时拿走你的文件（返回结果）

## 每个业务模块长什么样？

项目里每个功能模块（比如用户管理、角色管理）都遵循**同一个结构**：

```
module_user/
├── controller.py   ← 接口层：定义 URL 和请求方式（GET/POST/PUT/DELETE）
├── service.py      ← 业务层：写业务逻辑（比如"创建用户前先检查用户名是否存在"）
├── crud.py         ← 数据层：增删改查数据库（继承公共基类，大部分方法不用自己写）
├── model.py        ← 模型层：定义数据库表有哪些字段
└── schema.py       ← 数据校验层：定义请求/响应的数据格式（用 Pydantic）
```

**数据流向**：`请求 → controller → service → crud → 数据库 → 原路返回`

**新手开发流程**：当你需要开发一个新功能（比如"公告管理"）：
1. `model.py` — 定义"公告"有哪些字段（标题、内容、发布时间...）
2. `schema.py` — 定义创建公告时要传哪些参数、返回哪些字段
3. `crud.py` — 继承 `CRUDBase`，大部分增删改查已经有了，特殊的自己加
4. `service.py` — 写业务逻辑（比如"发布前检查标题不能为空"）
5. `controller.py` — 定义接口路径（`POST /api/v1/system/notice`），调用 service

## 权限体系（最复杂的部分，慢慢理解）

权限分两层：

### 第一层：功能权限 —— "你能不能用这个功能？"
- 每个用户 → 绑定若干角色（比如"管理员"、"普通员工"）
- 每个角色 → 绑定若干菜单/按钮权限（比如"用户管理页面"、"删除用户按钮"）
- 请求进来时，检查你的角色有没有对应权限标识

### 第二层：数据权限 —— "你能看到哪些数据？"
即使你都能"查看用户列表"，不同人看到的数据范围不同：

| 策略 | 你能看到什么 | 举例 |
|------|------------|------|
| 仅本人 | 只有自己的数据 | 只能看自己创建的订单 |
| 本部门 | 本部门所有人的数据 | 销售部的人看销售部的 |
| 本部门及以下 | 本部门+子部门 | 总部能看到分公司的 |
| 全部数据 | 所有数据 | 超级管理员 |
| 自定义 | 指定部门的数据 | 跨部门协作的特殊角色 |

**超级管理员**不受任何限制。

## 插件系统 —— 不改主代码也能加功能

项目用了一个很巧妙的设计：**插件自动发现**。

规则很简单：
1. 在 `app/plugin/` 下新建 `module_xxx/` 目录
2. 里面放 `controller.py`，定义一个 `APIRouter`
3. 启动时框架自动扫描并注册，路由前缀就是 `/xxx`

现有的插件：
- **module_ai** — AI 聊天功能
- **module_task** — 定时任务和工作流
- **module_generator** — 代码生成器（根据数据库表自动生成代码）
- **module_example** — 示例插件，新人的参考模板

**建议**：先看 `module_example`，它是专门写给你参考的。

## 启动流程（项目跑起来时做了什么）

```
python main.py run --env=dev
        │
        ▼
  1. create_app()       → 创建 FastAPI 应用
  2. 注册异常处理器      → 出错了统一格式返回
  3. 注册中间件         → CORS、日志、压缩
  4. 注册路由           → API 接口 + 插件自动发现
  5. lifespan 启动：
     a. 建表 + 种子数据  → 数据库初始化
     b. 连接 Redis       → 缓存服务就绪
     c. 加载系统配置到 Redis
     d. 加载数据字典到 Redis
     e. 启动定时任务调度器
     f. 启动请求限流器
  6. Uvicorn 开始监听端口 8001
        │
        ▼
  项目跑起来了！访问 http://localhost:8001/docs 可以看到所有接口文档
```

## 数据库相关

### 多数据库支持
项目支持三种数据库，通过配置切换：
- **MySQL** — 生产环境推荐
- **PostgreSQL** — 也可以
- **SQLite** — 开发测试用，不用额外安装

### 软删除
数据不会真正从数据库删掉，而是标记 `is_deleted = True`。这样误删了还能恢复。

### 数据库迁移
当你修改了 `model.py`（比如加了一个字段），需要：
```bash
python main.py revision --env=dev   # 生成迁移脚本（记录改动）
python main.py upgrade --env=dev    # 执行迁移（真正改数据库）
```

## Redis 在项目中的角色

Redis 在这里主要做**三件事**：

| 用途 | 说明 |
|------|------|
| 存登录令牌 | 用户登录后 token 存 Redis，请求时去 Redis 验证。过期了就自动失效 |
| 缓存配置和字典 | 系统配置、数据字典这些不常变的数据，从数据库读一次存 Redis，之后直接从 Redis 取，快得多 |
| 限流计数 | 记录你短时间内请求了多少次，超过限制就拒绝 |

## 配置管理

配置不硬编码在代码里，而是通过 `.env` 文件管理：

- `env/.env.dev` — 开发环境配置（数据库用本地、日志级别 DEBUG）
- `env/.env.prod` — 生产环境配置（数据库用正式服务器、日志级别 WARNING）

**核心原则**：敏感信息（数据库密码、JWT 密钥）永远不要写进代码，写在 `.env` 文件里，并且 `.env` 文件不要提交到 Git。

## 新人上手建议

### 第一步：把项目跑起来
1. 安装 Python 3.10+
2. 安装依赖：`pip install -r requirements.txt`
3. 复制 `env/.env.dev.example` 为 `env/.env.dev`，修改数据库和 Redis 配置
4. 运行：`python main.py run --env=dev`
5. 访问 `http://localhost:8001/docs` 查看接口文档

### 第二步：读代码顺序
1. `main.py` — 理解入口
2. `app/scripts/init_app.py` — 理解启动流程
3. `app/core/database.py` — 理解数据库连接
4. `app/api/v1/module_system/auth/` — 理解登录流程（最常用）
5. `app/plugin/module_example/` — 理解插件结构
6. 任意一个模块（比如 `user`）— 理解完整的 controller→service→crud 链路

### 第三步：动手写
- 照着 `module_example` 写一个简单的 CRUD 插件
- 试着加一个新字段到现有模型，跑一次 migration

### 遇到不懂的？
- FastAPI 基础 → 官方教程 https://fastapi.tiangolo.com/zh/
- SQLAlchemy 基础 → 官方教程 https://docs.sqlalchemy.org/
- Pydantic 基础 → 官方文档 https://docs.pydantic.dev/
- 异步编程 → 先理解 `async def` 和 `await` 就够了

## 术语速查表

| 项目中的词 | 你可以理解为 |
|-----------|------------|
| CRUDBase | 增删改查的"模板"，继承它就自动有了基本的增删改查方法 |
| lifespan | 应用的"生命周期"，启动时做什么、关闭时做什么 |
| 中间件 (Middleware) | 请求处理的"中间关卡"，每个请求都会经过 |
| 依赖注入 (Dependency) | FastAPI 的设计，让函数自动获取它需要的对象（比如数据库连接） |
| 种子数据 (Seed Data) | 系统第一次启动时写入的初始数据（比如默认管理员账号） |
| 软删除 | 不真删数据，只是标记为"已删除" |
| 滑动过期 | 用户在操作时，Token 的过期时间自动延长（就像商场停车，每消费一次就延长免费时间） |
| RBAC | 基于角色的权限控制，用户→角色→权限 |
| 多租户 | 一套系统服务多个客户，每个客户只能看到自己的数据 |
| 数据权限 | 同一个功能，不同人能看到的数据范围不同 |
